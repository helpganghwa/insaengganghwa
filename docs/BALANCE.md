## 1. 강화 — 사이클 시간 곡선 & 3분기 outcome 확률

### 1.0 사이클 (cycle) 개념

강화는 **100단위 사이클**로 진행. 레벨 L에 대해:

- `cycle = floor(L / 100)`, `cycleLevel ℓ = L mod 100` (∈ 0~99)
- **시도 시간**은 사이클마다 **2배**씩 증가: `d(L) = d₀(ℓ) × 2^cycle`
- **확률 곡선**은 사이클마다 리셋 — 성공률·하락률은 ℓ만의 함수
- 사이클 무한 진행(상한 없음). 진행할수록 per-attempt 시간만 지수 증가

**예**

| L | cycle | ℓ | 시간 배수 | 공시 성공률 | 하락률 |
|---|-------|---|-----------|-------------|--------|
| +50 | 0 | 50 | ×1 | 51% | 0 |
| +99 | 0 | 99 | ×1 | 25% | 15% |
| +100 | 1 | 0 | ×2 | 100% | 0 |
| +152 | 1 | 52 | ×2 | 48% | 8% |
| +199 | 1 | 99 | ×2 | 25% | 15% |
| +252 | 2 | 52 | ×4 | 48% | 8% |

### 1.1 시간 곡선 — d₀(ℓ) piecewise 선형 (사이클 0 기본)

- 베이스/카탈로그/슬롯 **전부 무관** — 모든 장비 동일 (GDD §3.2)

**1회 시도 소요 d₀(ℓ) — piecewise 선형 (초반 온보딩 가속, balance.ts):**

```
ℓ ∈ [0, 10]  : D_MIN(10s)   → D_AT_10(10m)  선형
ℓ ∈ [10, 99] : D_AT_10(10m) → D_MAX(215m)   선형
D_MIN = 10s  ·  D_AT_10 = 10m  ·  D_MAX = 215m
```

| ℓ | d₀(ℓ) | ℓ | d₀(ℓ) |
|---|-------|---|-------|
| +0 | 10s | +50 | ≈1.70h (102m) |
| +10 | 10m | +60 | ≈2.08h (125m) |
| +20 | ≈33m | +75 | ≈2.66h (160m) |
| +30 | ≈56m | +90 | ≈3.24h (194m) |
| +40 | ≈1.32h (79m) | +99 | **3.58h (215m)** |

사이클 1+ d(L) = d₀(ℓ) × 2^cycle.

> ⚠ 위 d₀(ℓ) 표는 **스케일 적용 전 base** 값이다. 2026-05-31 `ATTEMPT_DURATION_SCALE = 0.5`(per-attempt ÷2) 적용으로 **실제 1회 시도 시간은 표값의 절반**. 아래 누적 도달 시간은 스케일 적용 후 값.

**누적 도달 (사이클 0 기준 — 해석적 평균, full-wait, 스케일 적용 후):**

| 도달 레벨 | +0 → 누적 평균 시간 | 설계 의도 |
|-----------|---------------------|-----------|
| +30 | ≈ 11시간 | 1일 이내 |
| +50 | ≈ 1.6일 (37h) | 3일 이내 |
| +99 | ≈ **14일 (327h)** | **2주** |

> +30·+50의 design intent는 1일·3일이었으나, 위험구간 진입 전(ℓ≤51) drift=+이라 자연히 빨라짐(허용). +99의 14일(2주)이 핵심 제약(코드 앵커 `CUMULATIVE_REACH_ANCHORS_MS`).

**사이클별 누적 (이론값 = cycle0 × 2^c, 사이클 내 dynamics 동일):**

| 도달 레벨 | 누적 평균 시간 |
|-----------|----------------|
| +99 | ≈ 14일 (2주) |
| +199 | ≈ 42일 (6주) |
| +299 | ≈ 98일 (14주) |
| +399 | ≈ 210일 |

> 정확한 평균은 `scripts/analyze-enhance.ts`(해석적 expected first-passage time, ms). `scripts/tune-linear-duration.ts`로 D_MAX 재튜닝 가능. 결과는 `/admin/balance-review`에서 확인.

- 강화는 **순수 시간 + RNG** — 자원·제물 비용 없음. 카탈로그당 단일 강화레벨(`user_equipment`).
- idle 친화: 누적이 길어도 떠나 있는 동안 진행. 보석 단축으로 압축

### 1.2 시간 비례 4분기 outcome 확률 (대성공·성공·하락·유지)

```
p_success_total(L, elapsed, total) = baseRate(ℓ) × clamp(elapsed / total, 0, 1)
p_mega   (...)               = ⌊ p_success_total × MEGA_OF_SUCCESS_BP / 10000 ⌋  ← 성공 분량의 5% (+2 대성공)
p_success(...)               = p_success_total − p_mega                          ← 나머지 성공 (+1)
p_down   (L)                 = downRate(ℓ)            ← 시간 무관 고정
p_hold   (L, elapsed, total) = 10000 − p_success_total − p_down  (bp)
```

- `MEGA_OF_SUCCESS_BP = 500`(5%) — **성공 시 5% 확률로 2단계 상승(+2, 대성공)**, 나머지 95%는 +1. 성공 *총량*은 불변(메가는 성공 분량을 +1/+2로 쪼갠 것뿐)이라 하락·유지 확률엔 영향 없음.
- `elapsed` = 큐 등록 후 서버 시계 경과 (보석 단축분 포함), `total` = 등록 시점 산정된 총 소요 시간
- 큐 **완료**(elapsed ≥ total) 시 `p_success_total = baseRate(ℓ)` (공시 최대치)
- **하락 확률은 시간에 무관하게 고정** — 일찍 시도해도 동일 down%. 일찍 시도하면 잃은 success는 전부 **유지(hold)** 로 이동
- 시도 결과: 한 번의 RNG roll을 [0, p_mega) → 대성공(+2), [p_mega, p_mega+p_success) → 성공(+1), 다음 p_down 구간 → 하락, 나머지 → 유지로 분기 (resolve.ts 순서)
- 불변식: 모든 ℓ에서 `baseRate(ℓ) + downRate(ℓ) ≤ 10000` → `p_hold ≥ 0`
- RNG는 **시도 시점 서버**에서 `crypto` 기반 결정 (CLAUDE §3.1)

### 1.3 baseRate(ℓ) & downRate(ℓ) (사이클 내 동일, 채택값 2026-05-25)

`baseRate(ℓ)` = **끝까지 기다렸을 때의 공시 성공률**. `downRate(ℓ)` = **시간 무관 고정 하락 확률**.
고레벨 success를 상향하고 down은 독립 곡선으로 — 모든 ℓ에서 drift(up−down) > 0 →
사이클 0 +99 도달 평균 ≈ 14일 (2주 목표 1.00×, `scripts/analyze-enhance.ts`).

보간은 **앵커 간 선형**(`lerpAnchors`)이고, 아래 표의 ℓ이 코드 정본(`balance.ts`
`BASE_RATE_ANCHORS`/`DOWN_RATE_ANCHORS`)의 **앵커 전체**다. 특히 10→51 구간은 단일
직선이 아니라 **20:85% · 30:70% · 40:58%** 를 경유한다(예: +30 = 70%, 양끝 직선
보간값 75.6%가 아님). 앵커 사이 값은 인접 앵커의 선형 보간, 결과는 만분율 반올림.

| ℓ 앵커 | baseRate (max) | downRate | 유지 (max-t, =100−base−down) |
|--------|----------------|-----------|-------------------------------|
| 0 → 10 | 100% | 0% | 0% |
| 20 | 85% | 0% | 15% |
| 30 | 70% | 0% | 30% |
| 40 | 58% | 0% | 42% |
| 51 | 50% | 0% | 50% |
| 52 | 48% | 8% | 44% |
| 60 | 42% | 12% | 46% |
| 75 | 40% | 15% | 45% |
| 90 | 30% | 15% | 55% |
| 99 | 25% | 15% | 60% |

(모든 ℓ에서 drift(up−down) > 0 유지.)

**하락 규칙**
- 하락은 **ℓ ≥ 52에서 fixed downRate(ℓ) 확률로 발생**, 폭 = **−1 레벨** (단순·예측가능, 파괴 없음)
- 하락 하한 = **사이클 내 +51** (예: +152→+151 OK / +100→+99 불가 — +100은 안전구간이라 down 자체 0%)
- 자동 재등록: 성공 시 다음 레벨 자동 큐잉 / 실패(유지·하락) 시 중단 + 결과 모달 (GDD §3.2)

**설계 의도**
- 사이클 내 안전 구간(ℓ 0~51)은 down=0% — 실패해도 단계 유지(시간만 손해)
- 사이클 내 위험 구간(ℓ 52~99)은 −1 손실 위험 + 낮은 baseRate + early attempt 도박성 → 보석 단축 동기
- 사이클 경계(+100, +200, …)에서 확률은 리셋되지만 **시간이 2배씩 늘어** 누적 도달 시간은 지수 증가
- 구현: ℓ→bp(만분율) **단조 표**를 `lib/game/balance.ts`에 박제, 본 표와 1:1 (게임산업법 §33 공시 일치)

## 2. 초월 — 중복 수 & 전투력 % 배수

초월: 박스로 **같은 카탈로그 중복**을 모으면 `user_equipment.transcend_progress` 누적 → 임계 도달 시
**자동** `transcend_level +1`. **상한 없이 무한 진행**·무RNG (GDD §3.3).

### 2.1 레벨업 중복 수 — 선형(T단계 = T개)

T단계 달성에 같은 카탈로그 중복 **T개** 필요. 상한 없음 — 누적 = T(T+1)/2.

| 초월 단계 | 해당 단계 중복 수 | 누적 |
|-----------|------------------|------|
| 0→1 | 1 | 1 |
| 1→2 | 2 | 3 |
| 2→3 | 3 | 6 |
| 3→4 | 4 | 10 |
| 4→5 | 5 | 15 |
| 5→6 | 6 | 21 |
| 6→7 | 7 | 28 |
| 7→8 | 8 | 36 |
| 8→9 | 9 | 45 |
| 9→10 | 10 | 55 |
| 10→11 | 11 | 66 |
| … | … | … |
| 19→20 | 20 | 210 |

- T10까지 누적 **같은 아이템 55개**(대상 본체 별도). 이후로도 단계마다 선형 증가.
- 보급 상자는 슬롯 내 균등 랜덤이라 특정 아이템 중복 수집 = 장기 엔드게임 목표 (의도된 병목, GDD §3.4)

### 2.2 전투력 % 배수 — T0~T10 가속 곡선 + T11~ 레벨당 +10%p

`개별 장비 전투력 = (강화 기반 전투력) × (1 + transcendBonus[T])`

| T | transcendBonus | T | transcendBonus |
|---|----------------|---|----------------|
| 0 | +0% | 6 | +45% |
| 1 | +5% | 7 | +56% |
| 2 | +11% | 8 | +68% |
| 3 | +18% | 9 | +82% |
| 4 | +26% | 10 | **+100% (×2.0)** |
| 5 | +35% | 11~ | 레벨당 **+10%p** |

- T0~T10은 단계 이득 **가속** (GDD §3.3) — T10 = 그 장비의 강화 전투력을 2배로.
- **T11부터는 레벨당 +10%p 선형 무한 증가**(T20=+200%, T50=+500% …).
- 구현: `transcendFodderForStep(T)`, `transcendBonusBp(T)`를 `lib/game/balance.ts`에 박제, 본 절과 1:1

## 3. 전투력 (추천값)

### 3.1 강화 레벨 → 전투력 P(L)

모든 카탈로그 아이템 **동일 곡선** (베이스 차등 없음, GDD §3.1).

추천식: `P(L) = round(10 × (1 + L)^1.5)` — 초선형(지수 1.5)이라 고강화 투자가 전투력으로 확실히 보상.

| L | P(L) | L | P(L) |
|---|------|---|------|
| 0 | 10 | 50 | 3,642 |
| 5 | 147 | 51 | 3,750 |
| 10 | 365 | 60 | 4,764 |
| 20 | 962 | 70 | 5,983 |
| 30 | 1,726 | 80 | 7,290 |
| 40 | 2,625 | 99 | 10,000 |

(계수 10·지수 1.5는 추천 시작값 — 곡선 가팔기는 튜닝 가능)

### 3.2 개별 / 총 전투력

- **개별 장비 전투력** = `P(L) × (1 + transcendBonus[T])` (§2.2)
- **총 전투력** = **보유한 모든 카탈로그 아이템의 개별 전투력 합** (카탈로그당 1레코드)
  - **상태 기반**: 보유(`user_equipment` row) 카탈로그만 합산. 강화 하락 시 즉시 반영 → 강화 도박의 하락 리스크가 전투력에 그대로 작용
  - **착용 무관**: 착용 슬롯은 외형(프로필·자랑·OG 표시) 전용 — 전투력 가중 없음. **외형을 바꿔도 전투력은 변하지 않는다** (수집·외형 다양성 장려, GDD §3.1)
- 컬렉션 폭(보유 카탈로그 수) × 깊이(각 강화·초월)가 곧 전투력 — 별도 도감 계수 항 없이 합 자체로 보상
- 예: 무기 L50·T2(3,642×1.11≈4,044) + 방어구 L40·T0(2,625) + 장신구 L30·T1(1,726×1.05≈1,812) + 보유 활 L20·T0(962) → 총 전투력 = 4,044+2,625+1,812+962 = **9,443** (보유 카탈로그 모두 합산)

### 3.3 랭킹 지표 (GDD §5 — 시즌제 없음, 상시 누적)

> 세 지표 모두 **현재 보유 카탈로그 레코드** 기준(상태 기반) — 강화 하락 시 즉시 반영. lifetime(최고 도달)과 분리.

- **최고 강화** = 보유 카탈로그 중 단일 최고 강화 레벨 (max L)
- **합산 강화** = 보유 카탈로그의 강화 레벨 합 (Σ L)
- **전투력** = 위 총 전투력 (§3.2)

> 위 3개는 **전역(유저 단위)** 랭킹.

**아이템별 랭킹 (도감 내)** — catalog_item 1개당 별도 순위:
- 정렬: `max_enhance_level` DESC → `max_enhance_reached_at` ASC(먼저 달성) → `user_id` ASC(완전 결정성). Top10 노출
- **챔피언** = 해당 아이템 1위 (단 `max_enhance_level ≥ 1`; +0뿐이면 챔피언 없음). 동률이면 그 레벨을 **먼저 달성한 유저**
- 챔피언은 그 카탈로그 아이템의 모든 표시처(인벤토리·강화·자랑·공개프로필·OG)에서 스프라이트 챔피언 변형 노출(WIREFRAMES §7.2)
- **결정적 — 확률 없음**. 게임산업법 §33(확률형) 비대상. 시즌·리셋 없음(상시 누적)

> `P(L)` 표·초월 배수 표는 `lib/game/balance.ts`에 박제, 본 문서와 1:1.

## 4. 보급 (추천값)

### 4.1 카탈로그 (슬롯별, 가변)

- 카탈로그 아이템은 **슬롯(무기/방어구/장신구)별 분류**, 종수는 **고정 아님 — 지속 추가 예정** (GDD §10)
- 모든 아이템 성능 동일 (외관·도감·초월 동일성 판정용, GDD §3.1)
- 초기 예시: 무기 검/도끼/완드/활/스태프… · 방어구 갑옷/부츠/장갑/투구/방패… · 장신구 아뮬렛/귀걸이/팔찌/목걸이/반지…

### 4.2 박스 열기 확률

- 보급 상자는 **슬롯 일치** — 무기 상자→무기만, 방어구 상자→방어구만, 장신구 상자→장신구만
- 슬롯 내 **완전 균등**: `P(아이템) = 1 / (해당 슬롯 현재 카탈로그 종수)` — 종수가 늘면 확률 자동 재분배(항상 모두 동일 확률), **천장 없음**
- 획득 아이템 항상 **+0 · 초월0**
- 게임산업법 §33: 박스별 현재 아이템 목록·균등 확률을 게임 내 + `/probability` 공시 (카탈로그 추가 시 갱신·사전 고지)
- **공시 페이지 범위(§33은 확률형 한정)**: `/probability`는 *확률형* 요소만 공시 — §1 강화 성공률 · §2 초월(결정적이나 참고) · §3 전투력 · §4 보급 슬롯 균등 당첨. **레이드·재화/보상은 결정적(추첨 없음)이라 공시 페이지에서 제외**. 레이드·경제 *밸런스 스펙*은 본 문서 §5/§6에 유지(메커닉 운영 중) — 공시 의무 대상이 아닐 뿐.

### 4.3 중복 처리

- 박스로 같은 카탈로그가 또 나오면 **자동으로 초월 진행도에 누적**(§2). 별도 보관·정리 행위 없음 — 모든 중복이 곧 초월 연료.
- 보급 열기의 확률형 요소는 **슬롯 내 균등 당첨뿐**(추가 추첨 없음).

> 균등 확률 규칙은 `lib/game/balance.ts` 박제, 본 문서와 1:1. 카탈로그 종수는 가변이므로 확률은 "균등(1/현재 슬롯 종수)" **규칙**으로 박제(고정 수치 아님).

## 5. 레이드 (추천값)

> **재화 표기**: 다이아 ≡ 보석 — **단일 프리미엄 재화**(diamond). 강화 단축·레이드 개설/추가공격·IAP 모두 동일 재화 (상세 §6).

### 5.1 개설 / 참여 상수

| 항목 | 값 |
|------|----|
| 개설 비용 | **300 다이아** (즉시 차감, 환불 없음) |
| 최대 인원 | 10명 (호스트 포함) |
| 동시 진행 (유저당) | **3개** (호스팅+참여 합산) |
| 일일 진행 한도 (유저당) | **5회/일** (KST 기준) |
| 공격창 | 개설 후 **6시간** (서버 시계) |
| 기본 공격 횟수 | **10회 / 참여자** |
| 추가 공격 | 다이아로 구매 (§5.5) |

### 5.2 페이즈 HP

- phase1 HP = 랜덤 정수 `U(8000, 12000)`
- phase n HP = `phase1 × 1.5^(n-1)` (페이즈마다 ×1.5)
- 죽이는 개념 없음 — 6시간 만료로만 종료, **돌파 페이즈 수 = 보상 규모** (GDD §3.5)

### 5.3 1회 공격 데미지

```
dmg = round( 총전투력 × k × U(0.7, 1.3) × (크리 ? 1.5 : 1) )
```

- `k = 1.0` (추천 시작값 — 시뮬로 조정), 총전투력 = §3.2
- **미스 없음** (항상 명중) · 크리 확률 **5%** · 크리 배수 **×1.5** · 데미지 분산 **±30%** · **데미지 캡 없음**

### 5.4 보상 (전원 동일 — 기여도 무관)

1회 이상 공격한 참여자 **전원 동일** 지급(기본 참가 보상 없음 — 페이즈 돌파만 보상):

- **페이즈 돌파마다 보급 상자 1개** (전원 동일) — 슬롯 무작위(무기 / 방어구 / 장신구 각 **1/3** 균등). **다이아 드롭 없음**
- 기여도 가중 추첨 **없음** (GDD §3.5). 정산: 6시간 만료 시 lazy + cron 멱등 (CLAUDE §3.4)
- **수치 근거**: `scripts/simulate-raid-boxes.ts` — 페이즈당 1개는 무과금 패시브 박스 공급(≈675/월, §7.2+§6.4) 대비 보통 플레이 ×1.4~1.7·천장 ×3.3로 *보조* 수급에 안착. 1~3개는 헤비/고래 ×4~6.6로 초월 병목(§2.1) 붕괴 → 1개 고정 채택. (다이아 삭제만으로 기존 50%→100% 박스화로 박스 수율 이미 ~2배)

### 5.5 추가 공격 (다이아 구매)

- 기본 10회 소진 후 다이아로 추가 공격 구매 (해당 레이드 한정, 공격창 내)
- n번째 추가 공격 비용 (**10번 단위 계단 상승**): `25 × ⌈n / 10⌉` → 1~10번째 25, 11~20번째 50, 21~30번째 75, 31~40번째 100 …
- 추가 공격도 §5.3 동일 데미지 규칙

> 모든 수치 `lib/game/balance.ts` 박제, 본 문서와 1:1. `k`·크리·보상·추가공격 곡선은 시뮬/플레이테스트로 조정.

- **가격**: `9,900 + c×10,000`원, **상한 59,900원**(c5부터 평탄 — PG 등록 최고가 68,000원 이내). 강화·초월 동일.
