# SERVER — 서버 (논리 분리)

> 단일 DB 안에서 `server_id`로 게임 월드를 분리한다. **계정(카카오)은 전역, 캐릭터(게임 진행)는 서버별.**
> 물리 DB 분리 없이 "신서버 오픈"(새 사다리·새 영토)을 운영 이벤트로 제공 — 마이그레이션·백업·크론·어드민은 **×1 유지**(1인 운영 친화).

관련: `GDD.md`(코어 루프) · `SCHEMA.md`(스키마) · `GUILD.md`(점령전 — 서버별 월드) · `MELEE.md`(대난투 — 서버별 풀) · `REGULATORY` 관련은 §4(미성년 한도 중앙 집계)

---

## 0. 설계 원칙 (협상 불가)

- **단일 Postgres + 논리 분리** — 서버별 DB 분리 없음. 용량이 아니라 **콘텐츠 밀도·신규 사다리**를 위한 분리이므로 물리 분리가 필요 없다. (극단 스케일 시 `server_id` 기준 물리 분리로 승격 가능한 퇴로는 유지)
- **계정 = 전역 1개** — 카카오 인증·닉네임·본인인증·결제 기록·푸시 설정은 서버와 무관. 한 계정이 모든 서버에 캐릭터를 가질 수 있다.
- **캐릭터 = 서버별 진행** — 장비·강화·초월·보급·길드·점령·대난투·출석·우편 전부 서버 안에서만 존재.
- **다이아 = 서버별 지갑** — 서버 간 이전 불가(서버 경제 격리 — "전 서버 무료 파밍 → 본서버 집중" 어뷰징 차단). **결제 기록(`iap_orders`)·미성년 월 한도는 계정(중앙) 집계** — 게임산업법 한도는 명의 기준이므로 서버 무관 합산.
- **신서버 오픈 = 운영 이벤트** — `servers` 행 INSERT + 시드(zones 등). 코드 배포 불필요.

---

## 1. 전역 / 서버별 경계 (확정)

| 전역 (계정) | 서버별 (캐릭터) |
|---|---|
| 카카오 인증 · `profiles`(닉네임·publicCode·본인인증·푸시 설정·isAdmin) | `characters`(서버별 스칼라: 다이아 지갑·거주지·튜토리얼 단계) |
| **닉네임** — 계정당 1개(전 서버 동일, `/u/[nickname]` 유일성 유지) | 장비/강화/초월 (`user_equipment`·`enhancement_*`·`transcend_logs`·`gem_time_reductions`) |
| **AI 아바타 프로필** — 계정 자산(비용 들인 외형, 전 서버 공용) | 보급 (`supply_*`) · 도감 진행 |
| **친구** — 계정 단위(게임 정보 표시는 보는 서버의 캐릭터 기준) | 출석(`user_checkin_state`) · **우편**(보상이 서버 재화) |
| 결제 (`iap_orders`·`iap_refunds`·`identity_verifications`·월 한도) | 레이드 · 대난투(서버별 풀·서버별 챔피언) |
| 푸시 구독(기기 단위) · 신고(`*_reports`) · 추천(`referral_attributions`) | 길드 전부 · 점령전 · zones 소유/세금 상태 · **연대기**(서버별 역사) |
| `catalog_items`·`probability_snapshots`·`system_mode`(게임 컨텐츠/운영) | 배틀패스 **진행/수령**(구매 기록은 중앙, 적용 서버 명시) |
| | 리더보드 · 챔피언 · `shares`(생성 시점 서버 박제) |

**경계 디폴트 6** (구현 기준):
1. 닉네임 **전역** — 서버별 닉네임은 복잡도만 증가, 공개 프로필 URL 유일성 유지
2. AI 아바타 **전역** — 계정 자산. 대난투 리플레이 등 서버 화면에서도 그대로 사용
3. 친구 **전역** — 관계는 계정, 표시 데이터만 현재 서버 캐릭터
4. 출석·우편·튜토리얼 **서버별** — 보상이 서버 지갑으로 들어가므로
5. 공개 프로필/OG **서버 명시** — `/u/[nickname]?s=N`(기본 1), share 코드는 생성 서버 박제
6. 가입 보너스 **서버별 캐릭터 생성 시 지급** — 서버 경제가 격리라 다중 수령 무해(표준 MMO). 추천 보상은 공유가 발생한 서버 지갑으로

## 2. 데이터 모델

```sql
servers    (id smallint PK, name, status open|full|closed, opened_at)
characters (user_id FK profiles, server_id FK servers, diamond, residence_zone_id,
            tutorial_step, created_at, PK(user_id, server_id))
```

- **characters** = 서버별 스칼라 상태의 보관처(`profiles`의 서버 종속 컬럼이 단계 이관됨).
- **게임 테이블 스코핑** = 기존 `user_id` 키 유지 + **`server_id smallint not null default 1` 컬럼 추가** 방식(최소 침습 — 기존 행 자동 귀속, FK 재작성 없음). 유니크/인덱스는 `(server_id, …)` 복합으로 확장.
- 글로벌 단일 행 개념(대난투 일일 배틀, 일일 연대기, zones)은 `UNIQUE(server_id, …)`로 서버별 1행.

---

## 3. 컨텍스트 / 라우팅

- **활성 서버** = `srv` 쿠키(서버 선택 화면에서 설정, httpOnly) → `getActiveServerId()`. 미설정/비정상 = 1.
- 서버 액션·RSC는 `(userId, serverId)` 컨텍스트로 동작 — 기존 `getSessionUserId()` 옆에 serverId 헬퍼 추가, 도메인 함수는 serverId를 **명시 인자**로 받는다(크론과 시그니처 통일).
- **크론** = 활성 서버 루프: `for (const s of openServers) await run(s.id)` — 크론 엔트리 수는 불변(서버 수와 무관).
- **서버 선택 화면**: 로그인 후(또는 설정에서) 서버 목록 + 내 캐릭터 유무/전투력 표시. 다중 서버 오픈 전에는 화면 자체를 숨긴다(자동 1서버).

---

## 4. 결제 연계 (중앙 기록)

- `iap_orders`에 **`server_id` 기록**(지급 대상 지갑) — 구매 시점 활성 서버로 지급.
- **미성년 월 한도** = `iap_orders` **계정 합산**(서버 무관) — `monthly_purchase_limits`는 계정 키 유지.
- 환불 = 해당 주문의 server_id 지갑에서 회수(부족 시 마이너스 허용 + 안내 — GDD §3.8).
- 배틀패스 프리미엄 구매 = 중앙 주문 기록 + **적용 서버 명시**(서버별 진행에 귀속).

## 5. 구현 로드맵 (단계 커밋 · 단계 검증)

| 단계 | 내용 | 파급 |
|---|---|---|
| **P1 기반** | `servers`·`characters` 골격 + 1서버 시드 + 전 유저 백필 + `getActiveServerId()` | 무파괴(행동 변화 0) |
| **P2 지갑** | `profiles.diamond` → `characters.diamond` 이관 — 모든 차감/지급 경로 전환 | 전 도메인 다이아 참조 |
| **P3 코어** | 장비·강화·초월·보급·출석·우편 `server_id` 스코핑 + 복합 유니크 | 코어 쿼리 전반 |
| **P4 일일** | 레이드·대난투 스코핑 + 크론 서버 루프 | 크론·정산 |
| **P5 월드** | 길드·점령전·zones 상태·연대기 스코핑 | 길드 도메인 전체 |
| **P6 표면** | 서버 선택 UI·어드민 서버 관리·리더보드/OG/공개프로필 스코프·결제 server_id | UI·신서버 오픈 준비 |

- 각 단계: 마이그레이션(멱등) → 코드 전환 → `bun run build` + 테스트 → 커밋. 단계 사이에 배포 가능한 상태 유지.
- P2~P5 동안 기존 데이터는 전부 `server_id=1` — 유저 체감 변화 0.

## 6. 신서버 오픈 (운영)

- **오픈 트리거(권고)**: 1서버 점령전 포화(활성 길드가 구역 경합을 상시 채움) + DAU 임계(예: 5,000) 동시 충족. 단순 인구만으로 열지 않는다(인구 분산 = 콜드스타트 역행).
- 오픈 절차: `servers` INSERT → zones 시드(50구역) → 공지/이벤트(신서버 부스트). 코드 배포 불필요.
- `status=full` = 신규 캐릭터 생성 제한(기존 캐릭터는 정상) · `closed` = 준비 중/통합 대비.
- 서버 통합(인구 감소 시)은 v2 과제 — characters PK가 (user_id, server_id)라 단순 UPDATE 통합은 충돌 처리 필요(별도 설계).
