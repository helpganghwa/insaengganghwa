# PWA Push Notification 분석 + 도입 결정

> 작성일: 2026-05-25
> 검토 배경: GDD §3.10 "웹 푸시 미도입" 결정의 재검토
> 결정: **도입 확정** — 강화완료(그룹화) / 레이드 종료 / 일일보급충전 3개 우선

---

## 1. 결론

**2026년 기준 PWA로 푸시 알림은 충분히 가능하며, 광고와 달리 어뷰징 리스크가 거의 없고, 인생강화의 핵심 컨셉(시간기반 idle)에 직결된다.** GDD §3.10의 "1인 운영 부담" 우려는 도입 시점 한정이며 운영 중 부담은 거의 0이다.

| 플랫폼 | PWA Push | 비고 |
|--------|---------|------|
| Android Chrome | 완전 지원 | 네이티브 앱 푸시와 거의 동일한 경험 |
| **iOS Safari** | iOS 16.4+ 지원 (2023.3~) | **"홈 화면에 추가"한 PWA만** 가능 |
| Desktop (Chrome/Edge/Firefox) | 완전 지원 | 윈도우/맥 알림 센터까지 |

---

## 2. 기술 스택

```
브라우저 (Service Worker)
    ↑
    │ Web Push Protocol (VAPID)
    │
Push Service (Google FCM / Apple APNS)
    ↑
    │ HTTPS POST + VAPID 서명
    │
게임 서버 (Next.js + Vercel Cron)
```

| 컴포넌트 | 선택 | 비용 |
|---------|------|------|
| VAPID 키 | `web-push` 라이브러리로 생성 | 무료, 영구 |
| Push routing | FCM(Android) / APNS(iOS) — 자동 라우팅 | 무료, 무제한 |
| 구독 저장소 | Supabase Postgres (`push_subscriptions` 테이블) | 기존 인프라 |
| 발송 트리거 | Vercel Cron(일일 보급) + 기존 강화/레이드 cron | 기존 인프라 |

---

## 3. 광고와 결정적 차이 — 어뷰징 리스크

| 항목 | 광고 보상 | 푸시 알림 |
|------|----------|----------|
| 어뷰저 동기 | 보상 farming | **없음** (자기 디바이스 알림이라 가치 0) |
| SSV 필요성 | 결정적 | 불필요 |
| 클라이언트 신뢰 | 신뢰 불가 → 우회 가능 | 신뢰 무관 (서버가 일방 발송) |
| 비용 | 광고 네트워크 수수료 | 무료(VAPID/FCM) |

푸시는 사용자가 자기 디바이스에 받는 알림이고, 보상이 결합되어 있지 않아 farming 동기가 0이다. 도입 시 보안 부담이 광고와는 차원이 다르다.

## 4. 인생강화에 유용한 알림

| 알림 시나리오 | 가치 | 빈도 | v1 도입? |
|--------------|------|------|----------|
| **강화 완료** | 핵심 재접속 트리거 — "+38→+39 성공!" | 일 3~10회/유저 | **✓ 도입** (그룹화) |
| 레이드 만료 임박 (1h 전) | 보상 정산 전 마지막 공격 유도 | 일 0~3회 | (P1 후속) |
| **레이드 종료** | 정산 결과 확인 유도 | 일 0~3회 | **✓ 도입** |
| 친구가 +99 도달 | 사회적 자극·자랑 답방 | 주 0~5회 | (P2 후속) |
| **일일 보급 충전** (KST 자정) | 일일 미션 reminder | 일 1회 | **✓ 도입** |
| 배틀 패스 임박 (시즌 종료 D-3) | 미수령 단계 환기 | 시즌당 1회 | (배틀패스 도입 후) |
| 공유 답방 (친구가 내 자랑 클릭) | 바이럴 강화 | 가변 | (P2 후속) |

> v1 = 강화완료(그룹화), 레이드 종료, 일일 보급충전 3종만 우선 개발. 나머지는 도입 효과 측정 후 단계 확장.

## 5. 운영 부담 — 1인 운영 가능

| 작업 | 1회성/반복 | 부담 |
|------|----------|------|
| VAPID 키 생성 + 환경변수 등록 | 1회성 | 5분 |
| Service Worker + 구독 로직 | 1회성 | 1~2일 |
| 알림 발송 helper + cron 연결 | 1회성 | 1일 |
| 사용자 구독 토글 UI | 1회성 | 반나절 |
| **운영 중** | 반복 | **거의 0** — 코드가 자동 발송 |

GDD §3.10에서 우려한 "1인 운영 부담"은 도입 시점만 부담이고 운영 중 부담은 거의 없음.

## 6. 도입 시 함정·완화

### 6.1 권한 요청 마찰
- 첫 방문에서 즉시 prompt = 70%+ 거부 → **절대 금지**
- **contextual prompt**: 첫 강화 큐 등록 직후 "강화 끝나면 알려드릴까요?" 자연스러운 시점에 요청
- 거부 후 7일간은 재요청하지 않음 (재요청 빈도 자체가 distrust 신호)

### 6.2 iOS의 "홈 화면 추가" 요구
- iOS Safari PWA 푸시 = 홈 화면 추가가 전제
- **iOS 가이드 모달**: 첫 푸시 권한 요청 직전에 "iOS는 먼저 홈 화면 추가가 필요합니다 (방법 안내)" 단계
- 한국 iOS 점유율 50%+ → 결정적 제약이지만 한 번 추가하면 PWA 설치율·유지율 모두 급상승하므로 손익 +

### 6.3 알림 피로도
- 6슬롯 × 여러 사이클 강화 = 알림 폭격 위험
- **그룹화 정책**: 사용자별로 **30분 throttle 윈도** — 한 윈도 내 완료된 강화는 1건의 묶음 알림으로 발송
  - 예: "강화 3건 완료 (+38, +52, +73) — 결과 확인"
  - 윈도 내 첫 완료 시점 + 30분 후 발송, 그 사이 추가 완료는 누적

### 6.4 청소년 알림 규제
- 한국 법규: 야간(21시~익일 8시) **광고성** 푸시는 별도 사전 동의 필요
- 강화 완료·레이드 종료·일일 보급 충전은 **트랜잭션 푸시**(서비스 이용에 직결) → 야간 제한 대상 아님
- 마케팅 푸시(이벤트·할인)는 별도 옵트인 카테고리로 분리(v1 미도입)

### 6.5 푸시 권한 자체가 신뢰의 자산
- 한국 모바일 유저는 푸시 권한을 게임에 주는 것에 보수적
- 첫 1회 도입 후 마케팅으로 활용하면 권한이 빠르게 회수됨
- v1은 **트랜잭션 푸시만** — 사용자가 게임 진행과 연결된 가치만 받음

## 7. v1 도입 범위 (확정)

### 7.1 도입할 푸시 3종

> **결정 갱신(2026-07-10)**: 이후 기능 추가로 실제 스코프는 **7종** — 아래 3종(토글 제어) +
> **프로필(아바타) 생성 완료** · **추천 보상 도착** · **대난투 결과 발표**(상시, 토글 없음) ·
> **어드민 운영 알림**(어드민 전용). 전부 트랜잭션성(이용 직결)이라 야간 제한 대상 아님 —
> 확장을 의도된 스코프로 확정한다(`lib/push/send.ts` kind 참조).

| 알림 | 트리거 | 페이로드 | 그룹화 |
|------|--------|----------|--------|
| **강화 완료** | resolve 시점(lazy + cron) | "강화 {N}건 완료 — 결과 확인" | **30분 throttle 윈도** |
| **레이드 종료** | 6h 만료 cron | "{보스} 레이드 종료 — 보상 확인" | 레이드당 1회 |
| **일일 보급 충전** | 매일 KST 00:00 cron | "오늘의 보급 상자가 도착했어요" | 일 1회 |

### 7.2 권한 요청 흐름

```
첫 강화 큐 등록 성공
    ↓
"강화 완료 알림 받기" contextual modal
    ↓                         ↓
   동의                        거부
    ↓                         ↓
[iOS는 추가 단계]           7일 후 재제안 가능
"홈 화면 추가 가이드"
    ↓
브라우저 권한 prompt
    ↓
구독 등록 → DB
```

### 7.3 알림 카테고리 토글

기존 `notification_preferences` 컬럼 확장:
- `push_enhance` (기본 ON)
- `push_raid` (기본 ON)
- `push_supply` (기본 ON)

설정 페이지에서 카테고리별 ON/OFF 가능. 카테고리 OFF 시 해당 푸시 발송 스킵.

### 7.4 v1 미포함 (후속 의제)

- 친구 +99 자랑 답방 푸시 (P2)
- 레이드 만료 임박 1시간 전 푸시 (P1)
- 배틀패스 시즌 종료 D-3 (배틀패스 도입 후)
- 마케팅 푸시 (별도 옵트인 + 카테고리)

## 9. 한 줄 결론

> 광고와 달리 푸시는 도입하지 않을 이유가 거의 없다. **무료·어뷰징 리스크 없음·운영 부담 낮음·retention 효과 2~3배.** v1은 트랜잭션 푸시 3종(강화완료 그룹화 / 레이드 종료 / 일일 보급)으로 한정하여 사용자 신뢰를 보호한 채 효과만 흡수한다.
